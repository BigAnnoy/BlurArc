# 删除文件性能优化计划

## 问题诊断

删除文件慢的根本原因：**每删除一个文件，都会重新读写整个 `.photo_organizer.json` 文件来清除 MD5 记录**。

在 `api_server.py` L1670-1696：
```python
for file_path in file_paths:  # 假设删除 100 个文件
    path.unlink()  # 删除文件
    
    # 问题：每个文件都要 读→改→写 整个 JSON 文件
    if album_root:
        records_file = album_root / '.photo_organizer.json'
        if records_file.exists():
            with open(records_file, 'r') as f:  # 读整个文件
                raw = json.load(f)
            # 修改记录
            with open(records_file, 'w') as f:  # 写整个文件
                json.dump(out_data, f)
```

**性能瓶颈**：删除 N 个文件 = 读取 N 次 JSON + 写入 N 次 JSON，每次都是完整的 IO 操作。

## 优化方案

### 1. 批量处理 MD5 记录清除（后端）

将循环内的逐文件 IO 改为批量一次性 IO：

**修改 `api_server.py` 的 `delete_files()` 函数**：
- 先删除所有文件
- 只读取一次 `.photo_organizer.json`
- 批量清除所有已删除文件的 MD5 记录
- 只写回一次 `.photo_organizer.json`

**预期效果**：IO 操作从 `2N` 次降低到 `2` 次（N=文件数）

### 2. 前端加载优化（前端）

`AlbumBrowser.js` L601-603 在删除后会调用 `loadAlbumStats()`，可能需要刷新整个目录树。

**修改 `AlbumBrowser.js` 的 `_deleteSelectedPhotos()`**：
- 删除后仅从本地缓存中移除已删除照片（无需重新请求目录树）
- 避免不必要的全局统计刷新

**预期效果**：减少不必要的全局刷新，提升 UI 响应速度
