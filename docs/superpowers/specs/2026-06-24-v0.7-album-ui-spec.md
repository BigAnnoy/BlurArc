## 0. 数据模型

v0.7 基于 v0.6 已有的 `Photo` / `Album` / `AlbumPhoto` 三张表，仅做字段补充，**不引入新表**。

### 0.1 现有表（已存在于 v0.6）

```python
# backend/database.py（节选）
class Photo(Base):
    __tablename__ = "photos"
    id = Column(Integer, primary_key=True, index=True)
    filename = Column(String(255), nullable=False)
    path = Column(Text, nullable=False, unique=True)
    size = Column(Integer, nullable=False)
    md5_hash = Column(String(32), index=True)
    media_date = Column(DateTime(timezone=True), index=True)
    file_type = Column(String(10), nullable=False)   # 'photo' | 'video'
    is_favorite = Column(Boolean, default=False)    # v0.6 已有

class Album(Base):
    __tablename__ = "albums"
    id = Column(Integer, primary_key=True, index=True)
    name = Column(String(100), nullable=False, unique=True)
    created_at = Column(DateTime(timezone=True), server_default=func.now())
    description = Column(Text)

class AlbumPhoto(Base):
    __tablename__ = "album_photos"
    album_id = Column(Integer, ForeignKey("albums.id"), primary_key=True)
    photo_id = Column(Integer, ForeignKey("photos.id"), primary_key=True)
```

### 0.2 v0.7 新增字段

| 表 | 字段 | 类型 | 默认 | 索引 | 用途 |
|----|------|------|------|------|------|
| `photos` | `favorited_at` | `DateTime(timezone=True)` | NULL | ✅ | 收藏时间，用于"最近 7/30 天/一年"过滤 |
| `albums` | `cover_photo_id` | `Integer, ForeignKey(photos.id)` | NULL | — | 相册封面（v0.7 用第一张照片，**字段先建但 UI 暂不实现选择**） |
| `album_photos` | `added_at` | `DateTime(timezone=True)` | `now()` | — | 加入相册时间，用于"按加入时间"排序 |

### 0.5 关联关系

```
Photo ──< AlbumPhoto >── Album
                            │
                            └─ cover_photo_id ──> Photo（自引用，可空）
```

一张照片可属于多个相册（多对多）；一个相册的封面来自关联照片中的一张（v0.7 取第一张）。

#### 0.5.1 删除级联行为（CASCADE）

| 动作 | 影响 | 实现 |
|------|------|------|
| 删 `album` | `album_photo` 中该 album_id 全部记录被 CASCADE 删除 | SQLAlchemy `cascade="all, delete-orphan"` |
| 删 `photo` | `album_photo` 中该 photo_id 全部记录被 CASCADE 删除 | 同上 |
| 删 `album_photo` 单条 | 仅删除单条关联记录 | 普通 DELETE |
| 删 `cover_photo_id` 引用的 photo | `album.cover_photo_id` SET NULL | DB FK constraint `ON DELETE SET NULL` |

**重要不变量**：
- ❌ 删 album 不会删 photo 本身（照片是"主库"，相册是"虚拟集合"）
- ❌ 删 photo 不会删 album 本身（相册保留，可能变成空相册）
- ✅ 删 cover_photo 时，album 保留但封面置空（UI 显示 placeholder 灰块）

#### 0.5.2 唯一性约束

| 字段 | 约束 | 错误码 |
|------|------|--------|
| `album.name` | UNIQUE（不区分大小写） | `ALBUM_NAME_DUPLICATE` |
| `album_photo(album_id, photo_id)` | PRIMARY KEY 复合主键 | — |
| `photo.path` | UNIQUE | `PHOTO_PATH_DUPLICATE`（导入预检时检查） |

**大小写处理**：MySQL/PostgreSQL 默认大小写敏感，SQLite 默认大小写不敏感。v0.7 使用 SQLite，相册名 `Family Album` 和 `family album` 视为重复。

## 1. Sector 总览

#### 2.7.1 选择模式行为细节

**触发**：
- Toolbar 右侧"选择"按钮（§2.2 右侧 5）
- 快捷键：`Ctrl+A`（Windows）/ `Cmd+A`（macOS）— v0.7 仅 Windows
- PhotoCard 右上角悬浮 `☑` 按钮

**退出**：
- Esc 键
- Banner "取消" 按钮
- 切换 sector（自动清空 selectedIds）

**计数与限制**：

| 限制 | 值 | 行为 |
|------|------|------|
| 单次最大选中 | `1000` 张 | 超过时第 1001 张点击无响应 + toast `最多选中 1000 张` |
| 拖拽最大数 | `100` 张（见 §6.9.2）| — |
| 选中记忆 | **不持久化**（关闭 sector 即清空） | 避免大状态对象污染内存 |

**全选（Apple Photos 风格，二级全选）**：

| 步骤 | 触发 | 行为 | API |
|------|------|------|-----|
| 1. 一级全选 | 点 banner "全选" 按钮 | 选中**当前已加载的**照片（虚拟列表场景下未渲染的不选中） | — |
| 2. 显示二级入口 | 一级全选后 banner 自动追加 | 出现蓝色文字链接 `全选匹配项 (M 张)`，M = 当前 filter + view tab 下的总数 | `GET /api/photos/ids?{filters}` |
| 3. 二级全选 | 点二级入口 | selectedIds 扩展为完整 M 张 | 同上（id 列表仅 ~10KB，可一次性返回） |
| 4. 取消全选 | 点 "取消全选" | 清空 selectedIds | — |

**store 设计**（`frontend/src/stores/selectionStore.ts`，新增）：
```typescript
interface SelectionState {
  selectedIds: Set<number>;       // 当前选中的所有 photo_id
  totalMatching: number | null;   // 当前 filter 下总数（懒加载）
  isLoadingAllIds: boolean;       // 是否在拉完整 id 列表
  // 关键：所有 API 操作基于 selectedIds，与 UI 加载状态解耦
  // 例如：1000 张选中但只渲染 50 张 → "加入收藏" 仍操作 1000 张
}
```

**API 端点**（新增，§9.7）：
- `GET /api/photos/ids?favorite=true&type=photo&path=2024-10` → 返回 `{ ids: [1,2,3,...], total: 1234 }`
- 与 `GET /api/photos` 共用 query 参数，但**仅返回 id 字段**（不带 thumb/metadata，~10KB for 10K photos）
- 后端复用现有 SQL 过滤逻辑，加 `.pluck(:id)`

**UI 反馈**（banner 三态）：

```
状态 1: 普通选中
┌──────────────────────────────────────────────────┐
│ [已选 3 张]            [全选] [加入收藏] [删除] [取消] │
└──────────────────────────────────────────────────┘

状态 2: 一级全选后（显示二级入口）
┌──────────────────────────────────────────────────┐
│ [已选 1,234 张]  全选匹配项 (12,786 张)│
│                   [取消全选] [加入收藏] [删除] [取消] │
└──────────────────────────────────────────────────┘

状态 3: 二级全选进行中（spinner）
┌──────────────────────────────────────────────────┐
│ [已选 1,234 张]  正在加载全部 12,786 张... ⏳│
└──────────────────────────────────────────────────┘
```

**为什么需要二级**：
- 避免误操作（用户清楚知道"当前已加载 N" vs "匹配项总数 M"）
- 时间线场景下 12,786 张全选会让 UI 卡顿（即使不渲染，store Set 也会占用内存 ~100KB，可接受但要用户主动确认）
- 与 Apple Photos / Google Photos 行为一致，符合用户预期

**API 批量调用**：
- 后端 API 一次最多 500 ids（见 §9.5 per_page）
- 超过 500 张分批：例如 1200 张 → 3 次 POST，每次 500/500/200
- 失败分批容错：1 批失败不影响其他批
- 所有批次完成后 toast `已加入收藏 1200 张`（含跳过的去重计数）

#### 3.5.2 无限滚动实现

**方案**：IntersectionObserver 监听底部 sentinel，避免 scroll 事件频繁触发。

```typescript
// 前端伪代码
const sentinel = document.querySelector('.timeline-sentinel');
const observer = new IntersectionObserver(
  (entries) => {
    if (entries[0].isIntersecting && hasNext && !isLoading) {
      loadNextPage(cursor);
    }
  },
  { rootMargin: '400px 0px 400px 0px' }  // 提前 400px 触发
);
observer.observe(sentinel);
```

**触发条件**（全部满足才加载）：
1. sentinel 进入视口
2. `has_next=true`（API 响应字段）
3. `isLoading=false`（前次未完成）
4. 用户在底部 1000px 内（避免快速滚到顶就加载）

**性能**：
- 一次返回 12 个月分组（~ 1K-3K 照片）
- 渲染用 `react-window` 虚拟滚动（v0.7 仅时间线启用）
- 12 月/页 → ~ 200-500 DOM 节点，避免 React 重渲染

#### 3.5.3 "186 张照片" 计数 SQL

**单月计数**（已实现）：`SELECT COUNT(*) FROM photos WHERE YEAR(media_date)=Y AND MONTH(media_date)=M`

**分组计数**（API 返回）：
```sql
SELECT
  YEAR(media_date) AS year,
  MONTH(media_date) AS month,
  COUNT(*) AS photo_count
FROM photos
WHERE path = ?  -- 或 WHERE 1=1（时间线全局）
GROUP BY YEAR(media_date), MONTH(media_date)
ORDER BY year DESC, month DESC
LIMIT 12;
```

**索引要求**：`media_date`（§0.3 已建）。20K 照片时 < 100ms。

#### 3.5.4 异常处理

| 场景 | UI 反馈 |
|------|---------|
| 首次加载失败 | 主区域显示空态 `加载失败，点击重试` |
| 加载更多失败（已有数据） | 底部显示 inline 错误 `加载更多失败 [重试]`，不重置已加载数据 |
| API 超时（> 5s） | 同上 |
| hasNext=false | sentinel 隐藏，不再触发 |

### 6.8.1 Inline 重命名交互细节

**触发方式**（任一即可）：
- Toolbar "重命名" 按钮（§6.2 右侧 6）
- 标题 `家庭相册` 单击（变成 input）
- 主区域 AlbumCard 名称双击（§5.6）

**交互状态**：

| 状态 | 视觉 | 可触发动作 |
|------|------|------------|
| Idle | 普通文本 | 单击 / 双击 |
| Focus | `<input>` 替换文本，全选文字，蓝色边框 | 输入 / Enter / Esc / Blur |
| Saving | 右侧显示 loading spinner，input 禁用 | （等待 PATCH 响应） |
| Error | 红色边框 + 下方 inline 错误文字 | 重试 / Esc |
| Success | 短暂绿色 ✓ 0.5s → 回到 Idle | — |

**确认规则**：

| 触发 | 行为 |
|------|------|
| **Enter** | 提交修改（`PATCH /api/albums/{id}`） |
| **Esc** | 取消修改，恢复原名（不发起 API） |
| **Blur（点击外部）** | 等同 Enter，提交修改 |
| **连续按 Enter 间隔 < 300ms** | 忽略（防抖） |
| **空名 / 仅空格 / > 100 字** | 不提交，inline 错误 `名称不能为空` / `不能超过 100 字` |
| **重名（同其他相册）** | 不提交，inline 错误 `该名称已存在`（API 返回 409 ALBUM_NAME_DUPLICATE） |

**API 失败**（网络/5xx）：
- 不回滚 UI（避免标题闪烁）
- 显示 toast `重命名失败，请重试`
- input 重新变可编辑，光标定位末尾

**Loading 期间**（< 500ms 一般）：
- 不允许再次点击"重命名"按钮
- 不允许关闭当前 sector（提示 `保存中...`）

| PhotoCard | sidebar "⭐ 我的收藏" | 加入收藏（不离开相册） |
| PhotoCard（多选 N 张）| 任意 sector 的 AlbumCard | 批量加入 N 个相册 |

#### 6.9.1 拖拽视觉反馈

**Drag Source（被拖动的 PhotoCard）**：
- 透明度 0.4
- cursor: grabbing
- 拖动时显示 drag ghost（被拖 N 张时显示 `📷 N` 角标）

**Drop Target 状态机**：

| 状态 | 视觉 | 触发 |
|------|------|------|
| Normal | 默认 | 鼠标未进入目标 |
| Hover | 蓝色边框 2px + 浅蓝背景（`rgba(8,145,178,0.05)`） | dragenter |
| Active（可放下）| 绿色边框 2px + 绿色背景（`rgba(34,197,94,0.1)`） | dragover + dataTransfer.types 含 'photo' |
| Disabled | 红色边框 1px（`rgba(239,68,68,0.5)`） + 不允许 cursor | 目标相册已是当前相册 / 已是收藏 |
| Drop 成功 | 绿色 ✓ flash 0.3s | drop 触发 + API 200 |
| Drop 失败 | 红色 ✗ shake 0.3s + toast | API 4xx/5xx |

**dragover 必须 preventDefault**，否则 drop 不触发（HTML5 DnD 标准行为）。

#### 6.9.2 拖拽禁用场景

| 场景 | 反馈 |
|------|------|
| 拖到当前相册自身 | Disabled 态（避免重复加入） |
| 拖到"⭐ 我的收藏" 而照片已收藏 | Disabled 态 |
| 拖拽 > 100 张 | toast `单次最多拖动 100 张` + 阻止 dragstart |
| 目标相册正在被删除（API 进行中） | Disabled 态 |

#### 6.9.3 Drop API 行为

| 目标 | API | 成功后行为 |
|------|-----|------------|
| 另一个相册 | `POST /api/albums/{target_id}/photos {photo_ids:[]}` | 源相册内不删除（**v0.7 行为：复制，不移动**），详见下方 note |
| "⭐ 我的收藏" | `POST /api/photos/batch-favorite {photo_ids:[]}` | 已有 `is_favorite=true` 的跳过，toast `已加入收藏 (新增 N / 跳过 M)` |
| 收藏页"取消收藏"（反向）| `POST /api/photos/batch-unfavorite {photo_ids:[]}`（v0.7 不实现，反向操作走上下文菜单） | — |

> **v0.7 决策修正**：拖到相册是**复制**而非**移动**。
> - 原因：用户对"照片归属"心智模型更倾向"一张照片可以在多个相册"（相册是 tag/label，不是 folder）
> - 原 §6.9 表格中"移动到该相册（从当前相册移除并加入目标）"措辞不准确，**v0.7 改为"加入"（复制）**
> - 仍保留 §6.6 上下文菜单中的"从相册移除"用于主动移除

## 9. API 端点总览

### 9.1 收藏

| Method | Path | 说明 |
|--------|------|------|
| `POST` | `/api/photos/{id}/favorite` | 收藏 |
| `DELETE` | `/api/photos/{id}/favorite` | 取消收藏 |
| `POST` | `/api/photos/batch-favorite` | 批量收藏 `{photo_ids: [..]}` |
| `GET` | `/api/photos?favorite=true&page=N` | 列出收藏 |

### 9.2 相册

| Method | Path | 说明 |
|--------|------|------|
| `GET` | `/api/albums` | 列出所有相册（带 count） |
| `POST` | `/api/albums` | 创建相册 `{name, description}` |
| `PATCH` | `/api/albums/{id}` | 重命名 / 改描述 |
| `DELETE` | `/api/albums/{id}` | 删除相册（CASCADE album_photo） |
| `GET` | `/api/albums/{id}/photos?page=N` | 列出相册内照片 |
| `POST` | `/api/albums/{id}/photos` | 批量加入 `{photo_ids: [..]}` |
| `DELETE` | `/api/albums/{id}/photos/{photo_id}` | 单张移除 |
| `POST` | `/api/albums/{id}/photos/batch-remove` | 批量移除 `{photo_ids: [..]}` |

### 9.3 照片（已有 + v0.7 新增）

| Method | Path | 说明 |
|--------|------|------|
| `GET` | `/api/photos?path=2024-10&page=N` | 按目录列出 |
| `GET` | `/api/photos?favorite=true` | 列出收藏 |
| `GET` | `/api/photos?sort=date_desc&group_by=month` | 时间线分组 |
| `PATCH` | `/api/photos/{id}` | 修改 metadata（path, is_favorite 等） |
| `DELETE` | `/api/photos/{id}` | 删除照片 |
| `DELETE` | `/api/photos?ids=1,2,3` | 批量删除 |

### 9.4 文件夹（v1 + v0.7）

| Method | Path | 说明 |
|--------|------|------|
| `POST` | `/api/import/scan` | 增量扫描 `{path: xxx}` |
| `GET` | `/api/folders` | 列出物理文件夹（v0.7 暂未实现） |

### 9.5 分页参数（所有列表接口通用）

| 参数 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `page` | int | `1` | 页码，从 1 开始 |
| `per_page` | int | `100` | 每页数量，最大 `500` |
| `cursor` | string | — | 可选游标分页（用于时间线无限滚动，按 `media_date` DESC + `id` DESC 二元组） |

**响应格式**：
```json
{
  "items": [...],
  "total": 1234,
  "page": 1,
  "per_page": 100,
  "has_next": true
}
```

**时间线无限滚动**（§3.5）：
- 默认走 cursor 分页（`?cursor=2025-12-01T00:00:00_12345`）
- 每次返回 12 个月分组
- `has_next=false` 时停止加载

### 9.6 错误处理

所有 v0.7 新增 API 统一返回格式：

```json
{
  "error": {
    "code": "ALBUM_NAME_DUPLICATE",
    "message": "相册名称已存在",
    "details": {"existing_album_id": 5}
  }
}
```

| HTTP 状态码 | 场景 | code 示例 | UI 反馈 |
|------------|------|-----------|---------|
| `400` | 请求参数缺失 / 非法 | `INVALID_PARAM` | toast "请求参数错误" |
| `404` | 资源不存在（photo_id / album_id 无效） | `PHOTO_NOT_FOUND` | toast "照片不存在" + 刷新列表 |
| `409` | 相册名重复 | `ALBUM_NAME_DUPLICATE` | modal 内联错误，提示"该名称已存在" |
| `500` | DB 错误 / 文件系统错误 | `INTERNAL_ERROR` | toast "操作失败，请重试" + 错误日志 |
| `503` | 后台任务（扫描）进行中 | `IMPORT_IN_PROGRESS` | toast "导入进行中，请稍后再试" |

**前端统一拦截器**（`frontend/src/services/api.ts`）：
- 401 → 跳登录（v0.7 单机应用，理论上不出现）
- 404 → 从当前列表移除该 photo_id + toast
- 409 → 弹回 modal，不关弹窗
- 5xx → toast + Sentry 风格埋点（v0.7 简化为 `console.error` + toast）

### 9.7 仅返回 ID 的列表端点（v0.7 新增）

| Method | Path | 说明 |
|--------|------|------|
| `GET` | `/api/photos/ids` | 仅返回匹配条件下的 photo_id 列表（无 thumb/metadata） |

**用途**：实现二级全选（§2.7.1）。前端拿到 id 列表后扩展 `selectedIds`，避免虚拟列表场景下"全选"被截断。

**Query 参数**（与 `GET /api/photos` 完全一致）：

| 参数 | 类型 | 说明 |
|------|------|------|
| `favorite` | bool | 仅收藏 |
| `type` | string | `photo` / `video` |
| `path` | string | 物理路径（按年份浏览） |
| `screenshot` | bool | 仅截屏 |
| `edited` | bool | 仅已编辑 |
| `not_in_album` | bool | 不在相册中（LEFT JOIN） |
| `album_id` | int | 限定某相册内 |
| `from` / `to` | date | media_date 范围 |

**响应格式**：
```json
{
  "ids": [1, 5, 23, 87, ...],
  "total": 12786
}
```

**性能**：
- 10K id ≈ 50KB JSON（每 id ~5 字符 + 逗号）
- 不带 thumb → 不用走 thumbnail_manager
- 后端 SQL：`SELECT id FROM photos WHERE {filters} ORDER BY media_date DESC, id DESC`
- 索引：`media_date` + 各种 filter 字段（§0.3 已建）

**不使用 cursor 分页**：id 列表一次性返回（M ≤ 50,000 时 O(1) round-trip）。超过此规模后端截断 + 警告 header `X-Truncated: true`。

## 13. 跨章约束

### 13.1 平台支持

**v0.7 仅支持 Windows 桌面端**（PyWebView + Flask）。

涉及 Windows 专属 API 的功能：
- "在资源管理器中显示"：`subprocess.Popen(['explorer', '/select,', path])`
- FFmpeg 路径：`backend/ffmpeg_binaries/ffmpeg.exe`
- DB 路径：`项目根目录/.config/`（开发模式）或 `exe所在目录/.config/`（打包模式）

未来 Mac / Linux 适配时，这些功能需要重写或条件分支（v0.7 不预留抽象层）。

### 13.2 "导出"功能范围

v0.7 "导出..." 菜单仅支持 2 种粒度：

| 导出类型 | 范围 | 格式 | 实现 |
|---------|------|------|------|
| 导出原图 | 单张 / 批量 | 原始文件复制到目标目录 | 客户端 `shutil.copy2` + 选择目录 |
| 导出缩略图包 | 批量 | 缩略图打包为 ZIP | 复用 `thumbnail_manager` 现有缩略图 |

**不支持**（v0.7 范围外）：
- ❌ 视频导出
- ❌ 调色 / 滤镜 / 旋转后再导出
- ❌ HEIC → JPG 格式转换
- ❌ 邮件 / 网盘 / 第三方上传

### 13.3 性能预算

| 场景 | 目标 |
|------|------|
| 收藏视图首屏（10K 收藏） | < 500ms |
| 单个相册内照片列表（1K 照片） | < 300ms |
| "不在相册中" 过滤（20K 照片） | < 1s（**必须**用 LEFT JOIN，见 §4.4） |
| 时间线无限滚动每次加载 | < 800ms（12 月） |
| 加入收藏 / 取消收藏 API 响应 | < 100ms |
| 加入相册 modal 打开 | < 200ms |

### 13.4 国际化

v0.7 仅做**简体中文** UI（zh-CN）。
- 文案集中在 `frontend/src/i18n/zh-CN.ts`（新增文件）
- 不引入 i18n 框架（v0.7 直接 hardcode）
- 时间格式：`YYYY-MM-DD` / `YYYY年M月D日`
