## 0. 数据模型

v0.7 基于 v0.6 已有的 `Photo` / `Album` / `AlbumPhoto` 三张表，仅做字段补充，**不引入新表**。

### 0.1 现有表（已存在于 v0.6）

```python
# backend/database.py（节选）
class Photo(Base):
    __tablename__ = "photos"
    id = Column(Integer, primary_key=True, index=True)
    filename = Column(String(255), nullable=False)
    path = Column(Text, nullable=False, unique=True)
    size = Column(Integer, nullable=False)
    md5_hash = Column(String(32), index=True)
    media_date = Column(DateTime(timezone=True), index=True)
    file_type = Column(String(10), nullable=False)   # 'photo' | 'video'
    is_favorite = Column(Boolean, default=False)    # v0.6 已有

class Album(Base):
    __tablename__ = "albums"
    id = Column(Integer, primary_key=True, index=True)
    name = Column(String(100), nullable=False, unique=True)
    created_at = Column(DateTime(timezone=True), server_default=func.now())
    description = Column(Text)

class AlbumPhoto(Base):
    __tablename__ = "album_photos"
    album_id = Column(Integer, ForeignKey("albums.id"), primary_key=True)
    photo_id = Column(Integer, ForeignKey("photos.id"), primary_key=True)
```

### 0.2 v0.7 新增字段

| 表 | 字段 | 类型 | 默认 | 索引 | 用途 |
|----|------|------|------|------|------|
| `photos` | `favorited_at` | `DateTime(timezone=True)` | NULL | ✅ | 收藏时间，用于"按收藏时间排序"（§2.2 右侧 4 排序选项之一） |
| `albums` | `cover_photo_id` | `Integer, ForeignKey(photos.id)` | NULL | — | 相册封面（v0.7 用第一张照片，**字段先建但 UI 暂不实现选择**） |
| `album_photos` | `added_at` | `DateTime(timezone=True)` | `now()` | — | 加入相册时间，用于"按加入时间"排序 |
| `album_photos` | `sort_order` | `Integer` | `0` | — | 手动排序位置（v0.7 **字段先建但不实现拖动重排**，仅作为未来扩展点；UI 选"手动"时回退到"按最新"排序） |

### 0.5 关联关系

```
Photo ──< AlbumPhoto >── Album
                            │
                            └─ cover_photo_id ──> Photo（自引用，可空）
```

一张照片可属于多个相册（多对多）；一个相册的封面来自关联照片中的一张（v0.7 取第一张）。

#### 0.5.1 删除级联行为（CASCADE）

| 动作 | 影响 | 实现 |
|------|------|------|
| 删 `album` | `album_photo` 中该 album_id 全部记录被 CASCADE 删除 | SQLAlchemy `cascade="all, delete-orphan"` |
| 删 `photo` | `album_photo` 中该 photo_id 全部记录被 CASCADE 删除 | 同上 |
| 删 `album_photo` 单条 | 仅删除单条关联记录 | 普通 DELETE |
| 删 `cover_photo_id` 引用的 photo | `album.cover_photo_id` SET NULL | DB FK constraint `ON DELETE SET NULL` |

**重要不变量**：
- ❌ 删 album 不会删 photo 本身（照片是"主库"，相册是"虚拟集合"）
- ❌ 删 photo 不会删 album 本身（相册保留，可能变成空相册）
- ✅ 删 cover_photo 时，album 保留但封面置空（UI 显示默认封面：cyan 渐变 + 📷 + PHOTOS 标签，详见 §5.8）

#### 0.5.2 唯一性约束

| 字段 | 约束 | 错误码 |
|------|------|--------|
| `album.name` | UNIQUE（不区分大小写） | `ALBUM_NAME_DUPLICATE` |
| `album_photo(album_id, photo_id)` | PRIMARY KEY 复合主键 | — |
| `photo.path` | UNIQUE | `PHOTO_PATH_DUPLICATE`（导入预检时检查） |

**大小写处理**：MySQL/PostgreSQL 默认大小写敏感，SQLite 默认大小写不敏感。v0.7 使用 SQLite，相册名 `Family Album` 和 `family album` 视为重复。

## 1. Sector 总览

v0.7 在 v1 的 sidebar 基础上新增了 4 个 sector，**统一 3 层布局**：

```
<sector>
  ┌──────────────────────────────────────┐
  │ <toolbar>     标题 + 通用操作 + 视图操作 │  ← 第 1 层
  │ <view-tabs>   视图专属 tabs            │  ← 第 2 层
  │ <content>     photo-grid / albums-grid │  ← 第 3 层
  │               / timeline-view         │
  └──────────────────────────────────────┘
```

| 顺序 | Sector | 触发 | 数据源 | 主内容 |
|------|--------|------|--------|--------|
| 1 | **⭐ 收藏** | 点击 sidebar | `photo.is_favorite=1` | 照片网格 |
| 2 | **🕐 时间线** | 点击 sidebar | 全部照片 | All Photos / Months / Years 三视图 |
| 3 | **🖼 相册集** | 点击 sidebar | 全部相册 | 相册大卡片 |
| 4 | **📁 文件夹** | 点击 sidebar 任意子项 | 根目录 | **MainContent 照片网格**（基于 `state.selectedPath` 服务端查询）|

> **v0.7 变更**：原"按年份浏览"sector 已合并到时间线的 Years/Months 视图（§4.4.3-§4.4.4），不再作为独立 sector。

#### 2.7.1 选择模式行为细节

**触发**：
- Toolbar 右侧"选择"按钮（§2.2 右侧 5）
- 快捷键：不支持（v0.7 仅鼠标交互）
- PhotoCard 右上角悬浮 `☑` 按钮

**退出**：
- Esc 键
- Banner "取消" 按钮
- 切换 sector（自动清空 selectedIds）

**计数与限制**：

| 限制 | 值 | 行为 |
|------|------|------|
| 单次最大选中 | `1000` 张 | 超过时第 1001 张点击无响应 + toast `最多选中 1000 张` |
| 拖拽最大数 | `100` 张（见 §6.7.2）| — |
| 选中记忆 | **不持久化**（关闭 sector 即清空） | 避免大状态对象污染内存 |

**全选（Apple Photos 风格，二级全选）**：

| 步骤 | 触发 | 行为 | API |
|------|------|------|-----|
| 1. 一级全选 | 点 banner "全选" 按钮 | 选中**当前已加载的**照片（虚拟列表场景下未渲染的不选中） | — |
| 2. 显示二级入口 | 一级全选后 banner 自动追加 | 出现蓝色文字链接 `全选匹配项 (M 张)`，M = 当前 filter + view tab 下的总数 | `GET /api/photos/ids?{filters}` |
| 3. 二级全选 | 点二级入口 | selectedIds 扩展为完整 M 张 | 同上（id 列表仅 ~10KB，可一次性返回） |
| 4. 取消全选 | 点 "取消全选" | 清空 selectedIds | — |

**store 设计**（`frontend/src/stores/selectionStore.ts`，新增）：
```typescript
interface SelectionState {
  selectedIds: Set<number>;       // 当前选中的所有 photo_id
  totalMatching: number | null;   // 当前 filter 下总数（懒加载）
  isLoadingAllIds: boolean;       // 是否在拉完整 id 列表
  // 关键：所有 API 操作基于 selectedIds，与 UI 加载状态解耦
  // 例如：1000 张选中但只渲染 50 张 → "加入收藏" 仍操作 1000 张
}
```

**API 端点**（新增，§9.7）：
- `GET /api/photos/ids?favorite=true&type=photo&path=2024-10` → 返回 `{ ids: [1,2,3,...], total: 1234 }`
- 与 `GET /api/photos` 共用 query 参数，但**仅返回 id 字段**（不带 thumb/metadata，~10KB for 10K photos）
- 后端复用现有 SQL 过滤逻辑，加 `.pluck(:id)`

**UI 反馈**（banner 三态）：

```
状态 1: 普通选中
┌──────────────────────────────────────────────────┐
│ [已选 3 张]            [全选] [加入收藏] [删除] [取消] │
└──────────────────────────────────────────────────┘

状态 2: 一级全选后（显示二级入口）
┌──────────────────────────────────────────────────┐
│ [已选 1,234 张]  全选匹配项 (12,786 张)│
│                   [取消全选] [加入收藏] [删除] [取消] │
└──────────────────────────────────────────────────┘

状态 3: 二级全选进行中（spinner）
┌──────────────────────────────────────────────────┐
│ [已选 1,234 张]  正在加载全部 12,786 张... ⏳│
└──────────────────────────────────────────────────┘
```

**为什么需要二级**：
- 避免误操作（用户清楚知道"当前已加载 N" vs "匹配项总数 M"）
- 时间线场景下 12,786 张全选会让 UI 卡顿（即使不渲染，store Set 也会占用内存 ~100KB，可接受但要用户主动确认）
- 与 Apple Photos / Google Photos 行为一致，符合用户预期

**API 批量调用**：
- 后端 API 一次最多 500 ids（见 §9.6 per_page）
- 超过 500 张分批：例如 1200 张 → 3 次 POST，每次 500/500/200
- 失败分批容错：1 批失败不影响其他批
- 所有批次完成后 toast `已加入收藏 1200 张`（含跳过的去重计数）

### 2.8 拖拽支持

| 拖动源 | 拖动目标 | 行为 |

### 4.5 All Photos 无限滚动实现

#### 4.5.1 滚动机制

**方案**：IntersectionObserver 监听底部 sentinel。

```typescript
const sentinel = document.querySelector('.timeline-sentinel');
const observer = new IntersectionObserver(
  (entries) => {
    if (entries[0].isIntersecting && hasNext && !isLoading) {
      loadNextPage(cursor);
    }
  },
  { rootMargin: '400px 0px 400px 0px' }
);
observer.observe(sentinel);
```

**触发条件**（全部满足才加载）：
1. sentinel 进入视口
2. `has_next=true`（API 响应字段）
3. `isLoading=false`（前次未完成）

#### 3.5.2 Day 分组 SQL

```sql
-- 按天分组计数（API 返回，用于 section header）
SELECT
  DATE(media_date) AS day,
  COUNT(*) AS photo_count
FROM photos
WHERE {filters}
GROUP BY DATE(media_date)
ORDER BY day DESC
LIMIT 30;  -- 每次 30 天
```

**索引要求**：`media_date`（§0.3 已建）。20K 照片时 < 100ms。

#### 4.5.3 异常处理

| 场景 | UI 反馈 |
|------|---------|
| 首次加载失败 | 主区域显示空态 `加载失败，点击重试` |
| 加载更多失败（已有数据） | 底部显示 inline 错误 `加载更多失败 [重试]`，不重置已加载数据 |
| API 超时（> 5s） | 同上 |
| `has_next=false` | sentinel 隐藏，不再触发 |
| 概览图无数据 | 显示空态 `没有照片` |

### 4.6 概览图（Months/Years）实现

#### 4.6.1 概览图通用行为

- **无 sticky header**：整屏网格，竖向滚动
- **缩略图大小**：Zoom -/+ 控制（不改变网格结构，只改变缩略图尺寸）
- **空数据跳过**：0 张照片的月份/年份不渲染
- **代表性缩略图**：取该分组第一张照片（按当前排序）
- **无选中态**：概览图层级无"单击选中高亮"概念，双击直接进入 All Photos（带 filter，§4.6.2）

#### 4.6.2 概览图双击进入 All Photos

| 当前视图 | 双击行为 | 目标 |
|----------|----------|------|
| Years | 双击某年 | → All Photos（`?year=YYYY`，显示整年照片）|
| Months | 双击某月 | → All Photos（`?year=YYYY&month=M`，显示整月照片）|

**对齐 Apple Photos Tahoe**：双击概览卡片直接进入 All Photos（带时间 filter），不再逐级下钻。返回按钮回到来源概览。

#### 4.6.3 概览图分批加载

概览图卡片数量可能很大（大量月份/年份卡片），需分批加载避免一次性渲染卡顿。

**机制**：复用 §4.5 的 IntersectionObserver + sentinel 模式。

```typescript
// 概览图通用分批加载
const observer = new IntersectionObserver(
  (entries) => {
    if (entries[0].isIntersecting && hasNext && !isLoading) {
      loadNextBatch(cursor);  // 每次 30 个卡片
    }
  },
  { rootMargin: '400px 0px 400px 0px' }
);
```

**API 改动**（后端）：
- `/api/timeline/years` 加 `?limit=30&cursor=<year>` 参数
- `/api/timeline/months` 加 `?limit=30&cursor=<year-month>` 参数
- 响应加 `next_cursor` 和 `has_next` 字段

**性能预算**：每次 30 个卡片渲染 < 50ms，滚动平滑。

### 5.6 拖拽支持

| 拖动源 | 拖动目标 | 行为 | API |
|--------|----------|------|-----|
| AlbumCard | 另一个 AlbumCard | **合并相簿**：源相簿内所有照片加入目标相簿，源相簿保留（不删除） | `POST /api/albums/{target_id}/merge {source_id}` |
| PhotoCard | AlbumCard（任何 sector） | 加入该相册 | `POST /api/albums/{id}/photos` |

**合并相簿交互细节**：
- 拖动时目标 AlbumCard 显示绿色边框（drop active）
- drop 后弹确认 modal：`将「源相簿」的 N 张照片合并到「目标相簿」？源相簿将保留。`
- 按钮：`取消` / `合并`
- 重复照片自动去重（`INSERT OR IGNORE`）
- 合并成功 → toast `已合并 N 张照片（M 张已存在跳过）`

**API 响应格式**（`POST /api/albums/{target_id}/merge`）：

```json
{
  "target_album_id": 5,
  "source_album_id": 8,
  "added_count": 47,
  "skipped_count": 3
}
```

- `added_count`：实际新增到目标相册的照片数
- `skipped_count`：目标相册已存在跳过的数量（去重）

### 5.8 默认相册封面（empty/无封面相册）

> **设计目标**：v0.7 封面由 `cover_photo_id` 决定，但**空相册（无照片）**、**封面照片被删**、**封面照片未导入** 3 种情况都需要 fallback UI。
> 原型：[docs/prototypes/desktop/album-management-v2-light.html](../../prototypes/desktop/album-management-v2-light.html)

**触发场景**：

| 场景 | 条件 | 行为 |
|------|------|------|
| 空相册 | `album.cover_photo_id IS NULL AND COUNT(album_photo)=0` | 显示默认封面 |
| 封面被删 | 原 cover_photo 被删，DB SET NULL | 显示默认封面 |
| 封面未导入 | cover_photo 文件丢失 | 显示默认封面 |
| 加入相册 modal | 任何 album（包括已有照片）| 卡片均显示默认封面（统一风格） |

**默认封面视觉**（v3 设计，与整体 cyan 主色一致）：

```
┌──────────────────────┐
│                      │
│        📷           │  ← 24px emoji
│      PHOTOS         │  ← 10px 500 灰字
│                      │
└──────────────────────┘
```

- 背景：`linear-gradient(135deg, #f0f9ff 0%, #e0f2fe 100%)`（亮色）或 `linear-gradient(135deg, #0f172a 0%, #1e293b 100%)`（暗色）
- 叠加：`radial-gradient(circle at 30% 30%, rgba(34, 211, 238, 0.18) 0%, transparent 50%)` + `radial-gradient(circle at 70% 70%, rgba(8, 145, 178, 0.18) 0%, transparent 50%)`
- 📷 emoji 颜色：`var(--primary)`（亮色 #0891b2 / 暗色 #22d3ee）
- PHOTOS 文字颜色：`var(--text-tertiary)`，letter-spacing 0.5px
- 比例：3:2（与主页 photo card 一致）
- 圆角：6px（亮色与 photo card 同）

**CSS 复用**：

```css
/* 主区域 AlbumCard 用 */
.tile-default { ... }   /* 全填充 100%/100% */
/* Modal AlbumCard 用 */
.album-pick-card .cover-default { ... }   /* aspect-ratio: 3/2，padding 包裹 */
```

**应用位置**：

| 位置 | CSS class | 父容器 | 比例 |
|------|-----------|--------|------|
| §5.5 主区域 AlbumCard（无封面） | `.tile-default` | `.album-card` | 撑满（100% 100%）|
| §5.7 加入相册 modal 所有卡片 | `.cover-default` | `.album-pick-card` | 3:2 (aspect-ratio) |
| 未来：sidebar 缩略图 | `.thumb-default` | `.album-row` | 1:1 36×36 |

**为什么不用「拼色」方案**：v0.7 早期原型用过多色块拼贴当默认封面（仿 Apple Photos），但跟整体 cyan 主色调不符且视觉跳跃。v0.7.1 统一改为单色渐变 + 极简图标。

#### 6.6.1 Inline 重命名交互细节

**触发方式**（任一即可）：
- Toolbar "重命名" 按钮（§6.2 右侧 3）
- 标题 `家庭相册` 单击（变成 input）
- 主区域 AlbumCard 名称双击（§5.5）

**交互状态**：

| 状态 | 视觉 | 可触发动作 |
|------|------|------------|
| Idle | 普通文本 | 单击 / 双击 |
| Focus | `<input>` 替换文本，全选文字，蓝色边框 | 输入 / Enter / Esc / Blur |
| Saving | 右侧显示 loading spinner，input 禁用 | （等待 PATCH 响应） |
| Error | 红色边框 + 下方 inline 错误文字 | 重试 / Esc |
| Success | 短暂绿色 ✓ 0.5s → 回到 Idle | — |

**确认规则**：

| 触发 | 行为 |
|------|------|
| **Enter** | 提交修改（`PATCH /api/albums/{id}`） |
| **Esc** | 取消修改，恢复原名（不发起 API） |
| **Blur（点击外部）** | 等同 Enter，提交修改 |
| **连续按 Enter 间隔 < 300ms** | 忽略（防抖） |
| **空名 / 仅空格 / > 100 字** | 不提交，inline 错误 `名称不能为空` / `不能超过 100 字` |
| **重名（同其他相册）** | 不提交，inline 错误 `该名称已存在`（API 返回 409 ALBUM_NAME_DUPLICATE） |

**API 失败**（网络/5xx）：
- 不回滚 UI（避免标题闪烁）
- 显示 toast `重命名失败，请重试`
- input 重新变可编辑，光标定位末尾

**Loading 期间**（< 500ms 一般）：
- 不允许再次点击"重命名"按钮
- 不允许关闭当前 sector（提示 `保存中...`）

### 6.7 拖拽支持

| 拖动源 | 拖动目标 | 行为 |
|--------|----------|------|
| PhotoCard | sidebar 另一个相册 | 加入该相册（不从当前相册移除，相册是 tag 不是 folder） |
| PhotoCard | sidebar "⭐ 我的收藏" | 加入收藏（不离开相册） |
| PhotoCard（多选 N 张）| 任意 sector 的 AlbumCard | 批量加入 N 个相册 |

#### 6.7.1 拖拽视觉反馈

**Drag Source（被拖动的 PhotoCard）**：
- 透明度 0.4
- cursor: grabbing
- 拖动时显示 drag ghost（被拖 N 张时显示 `📷 N` 角标）

**Drop Target 状态机**：

| 状态 | 视觉 | 触发 |
|------|------|------|
| Normal | 默认 | 鼠标未进入目标 |
| Hover | 蓝色边框 2px + 浅蓝背景（`rgba(8,145,178,0.05)`） | dragenter |
| Active（可放下）| 绿色边框 2px + 绿色背景（`rgba(34,197,94,0.1)`） | dragover + dataTransfer.types 含 'photo' |
| Disabled | 红色边框 1px（`rgba(239,68,68,0.5)`） + 不允许 cursor | 目标相册已是当前相册 / 已是收藏 |
| Drop 成功 | 绿色 ✓ flash 0.3s | drop 触发 + API 200 |
| Drop 失败 | 红色 ✗ shake 0.3s + toast | API 4xx/5xx |

**dragover 必须 preventDefault**，否则 drop 不触发（HTML5 DnD 标准行为）。

#### 6.7.2 拖拽禁用场景

| 场景 | 反馈 |
|------|------|
| 拖到当前相册自身 | Disabled 态（避免重复加入） |
| 拖到"⭐ 我的收藏" 而照片已收藏 | Disabled 态 |
| 拖拽 > 100 张 | toast `单次最多拖动 100 张` + 阻止 dragstart |
| 目标相册正在被删除（API 进行中） | Disabled 态 |

#### 6.7.3 Drop API 行为

| 目标 | API | 成功后行为 |
|------|-----|------------|
| 另一个相册 | `POST /api/albums/{target_id}/photos {photo_ids:[]}` | 源相册内不删除（**v0.7 行为：复制，不移动**），详见下方 note |
| "⭐ 我的收藏" | `POST /api/photos/batch-favorite {photo_ids:[]}` | 已有 `is_favorite=true` 的跳过，toast `已加入收藏 (新增 N / 跳过 M)` |
| 收藏页"取消收藏"（反向）| `POST /api/photos/batch-unfavorite {photo_ids:[]}`（见 §9.1） | `updated_count: N, skipped_count: M` |

> **v0.7 决策**：拖到相册是**复制**而非**移动**。
> - 原因：用户对"照片归属"心智模型更倾向"一张照片可以在多个相册"（相册是 tag/label，不是 folder）
> - 仍保留 §6.4 上下文菜单中的"从相册移除"用于主动移除

### 7.5 为什么不复用 TimelineView

> **设计决策**：文件夹 sector 主区域用 `MainContent`，**不**复用 Sector 2 的 `TimelineView` 组件。

**原因**：
- `TimelineView` 内部用 `getTimelinePhotos` 拉全库照片再客户端 filter，**不支持**服务端按 `path` 查询
- 文件夹 sector 需要走 `getPhotos?path=xxx`（§9.3）做服务端 `WHERE path LIKE '/2024-10/%'` 过滤
- 全库拉到客户端再 filter 在 10K+ 照片库下首屏 > 2s，无法接受（§14.3 性能预算）

**未来重构路径**（v0.8+ 评估）：
- 若 TimelineView 增加 `?path` query param 支持，可合并两个 sector 复用同一组件
- v0.7 维持分离，避免改动 TimelineView 引入回归

### 9.7 错误处理

所有 v0.7 新增 API 统一返回格式：

```json
{
  "error": {
    "code": "ALBUM_NAME_DUPLICATE",
    "message": "相册名称已存在",
    "details": {"existing_album_id": 5}
  }
}
```

| HTTP 状态码 | 场景 | code 示例 | UI 反馈 |
|------------|------|-----------|---------|
| `400` | 请求参数缺失 / 非法 | `INVALID_PARAM` | toast "请求参数错误" |
| `404` | 资源不存在（photo_id / album_id 无效） | `PHOTO_NOT_FOUND` | toast "照片不存在" + 刷新列表 |
| `409` | 相册名重复 | `ALBUM_NAME_DUPLICATE` | modal 内联错误，提示"该名称已存在" |
| `500` | DB 错误 / 文件系统错误 | `INTERNAL_ERROR` | toast "操作失败，请重试" + 错误日志 |
| `503` | 后台任务（扫描）进行中 | `IMPORT_IN_PROGRESS` | toast "导入进行中，请稍后再试" |

**前端统一拦截器**（`frontend/src/services/api.ts`）：
- 401 → 跳登录（v0.7 单机应用，理论上不出现）
- 404 → 从当前列表移除该 photo_id + toast
- 409 → 弹回 modal，不关弹窗
- 5xx → toast + Sentry 风格埋点（v0.7 简化为 `console.error` + toast）

> §12 "范围外"之后列的 §12.1-12.4 实际是 v0.7 范围内的**补充设计**（XMP 同步、拍摄信息 chip 化、预览页 v2、数据目录迁移）。重命名为 §13 以消除命名冲突，原 §13、§14 → §14、§15。

### 13.1 v0.7 描述（XMP 同步机制）

> **v0.7 简化决策**：只实现"描述"一个文本字段，标签/人物/AI 标签全部不在 v0.7 范围。

**字段映射**：
| 应用字段 | XMP namespace | 备注 |
|----------|---------------|------|
| `photo.title` | `dc:title` | 标题 |
| `photo.description` | `dc:description` | 描述（用户笔记、回忆） |
| `photo.rating` | `xmp:Rating` | 0~5 星（可选） |

**同步策略**：
- **写入**：用户编辑描述 → 同步写两个位置
  1. **XMP**：直接写回原图（HEIC/JPG）—— 保证 Lightroom/Capture One/Apple Photos 都能读到
  2. **DB**：`photo.description` 字段（SQLite）—— 加快列表查询（`LIKE` 搜索）
- **读取**：打开照片时优先 DB（快），缺失时从 XMP 读（兼容外部修改）
- **冲突处理**：XMP 与 DB 不一致时，**以 XMP 为准**（外部编辑器是事实来源）
- **写入失败**：原图只读 / 格式不支持（PNG/PSD）→ 仅写 DB，UI 提示"原图为只读，仅保存到数据库"

**API 端点**：
```
GET    /api/photos/{id}              → 返回 title/description（从 DB）
PUT    /api/photos/{id}/description  → 同步写 XMP + DB
                                       { description: "..." }
```

**实现依赖**：
- `python-xmp-toolkit`（libxmp）or `pyexiv2`（更轻量）—— 读写 XMP
- HEIC 支持：`pyexiv2` + `libheif`
- 写入是 in-place 修改（原子性：先写 `.tmp` 再 rename）

**v0.7 UI 限制**：
- 预览面板：标题 + 描述（textarea，可编辑）
- 主视图 PhotoCard：只显示描述前 30 字符（hover 显示完整）
- 搜索：`LIKE '%xxx%'` 匹配 title/description（v0.7 不做全文检索）

> **为什么不上 DB-only？** 描述是用户对照片的"记忆标注"，换工具（Lightroom）编辑后必须能带回去；XMP 是行业标准。
> **为什么不上 XMP-only？** 全库搜索/排序需要 SQL 索引，XMP 需要重新解析每张图。

### 13.2 v0.7 拍摄信息面板（chip 化设计）

> **v0.7 决策**：拍摄信息用 **chip 标签** 呈现，**不用等宽数字字体显示日期**。统一字体链路：system-ui（中文用 PingFang/Microsoft YaHei，英文用 SF/Segoe），数字用 tabular-nums 等宽对齐。

**面板布局**：

```
┌─ 拍摄信息 ─────────────────────┐
│ 📅 2026 年 10 月 25 日 · 14:32  │  ← meta-row（日期行，system 字体）
│                                  │
│ [▭ 3 : 2] [⊞ 4032 × 3024]      │  ← meta-chips（等宽数字）
│ [💾 4.2 MB]                      │
└──────────────────────────────────┘
```

**字段**：

| 字段 | 数据来源 | 显示格式 | 字体 |
|------|----------|----------|------|
| 拍摄时间 | EXIF `DateTimeOriginal` | `YYYY 年 M 月 D 日 · HH:MM` | system |
| 宽高比 | `width / height` 化简 | `3 : 2` / `16 : 9` / `1 : 1` / `2 : 3` | monospace |
| 像素尺寸 | `width × height` | `4032 × 3024` | monospace |
| 文件大小 | 文件字节数 | `4.2 MB` / `512 KB` | monospace |

**Chip 样式规范**：
- 高 24px，水平 padding 9px，border-radius 12px（pill 形）
- 背景 `--bg-page`（浅灰），边框 `--border`
- 字号 12px，字重 500，字体 `SF Mono / Fira Code / monospace`，启用 `font-variant-numeric: tabular-nums`（数字等宽对齐）
- 每个 chip 配一个 12px 图标（轮廓线，宽 1.5px）
- hover 态：背景 `--primary-light`（青色淡底），边框/文字 `--primary`（青）
- 行间距 6px，自动换行

**字体统一原则**：
- ❌ **不要** 中文用 monospace（中文等宽字体会很丑，掉字距）
- ❌ **不要** 数字用衬线字体（数字宽窄不一，对齐差）
- ✅ **要** 中英文用 system 字体
- ✅ **要** chip 里的数字用 monospace + tabular-nums
- ✅ **要** 整个面板只用 2 种字体：system（文字行） + monospace（chip 数字）

**v0.7 隐藏字段**（v1 专业版有，v2 删除）：
- ❌ EXIF 12 项（相机/镜头/光圈/快门/ISO/焦距/色彩空间/方向/软件/经度/纬度/海拔）
- ❌ AI 建议 / 相似照片 / 地图
- ❌ 缩放指示器

> **为什么 chip 化？** "3 : 2 / 4032 × 3024 / 4.2 MB" 这三项是**多维属性**而非时间序列，chip 标签能让用户一眼区分；家用心智，零学习成本。

#### 12.3.7 视频预览支持（v0.7 新增）

> 预览页支持**视频**和**图片**同上下文混排，按 `file_type` 动态切换 DOM。
> 行为对齐 Apple Photos：进入自动播放 / 控件按需 / 放完自动跳。

##### 12.3.7.1 顶栏类型徽章

- 顶栏标题左侧显示**类型徽章**（与"位置"同行）：
  - 视频：青底 + 三角播放图标 + 文字"视频"
  - 照片：灰底 + 相机图标 + 文字"照片"
- CSS：`.preview-shell.is-video .type-badge.video { display: inline-flex; }`，反之显示 photo
- 字段渲染：`<span class="type-badge video">` / `<span class="type-badge photo">`

##### 12.3.7.2 主图区：图片 ↔ 视频动态切换

| 媒体 | DOM | 默认态 |
|------|-----|--------|
| 图片 | `<div class="main-image">` | 显示 |
| 视频 | `<video class="main-video" muted loop playsinline controls>` | `is-video` 时显示，否则 `display: none` |

**切换逻辑**：
- 后端返回 `photo.file_type`（`'photo' | 'video'`）
- 前端切到某张时 `shell.classList.toggle('is-video', isVideo)`
- CSS：`.preview-shell.is-video .main-image { display: none }` / `.preview-shell.is-video .main-video { display: block }`

**视频态行为**：
- `muted loop playsinline`：静音 + 循环 + 内联播放（WebView 友好）
- `controls`：原生控件始终存在，但**默认隐藏**（CSS `::-webkit-media-controls { display: none }`），鼠标 hover / 移动时显示，2.5s 无动作后自动隐藏
- 进入预览页：自动 `video.play()`，**静音**避免突然出声
- 离开预览页：自动 `video.pause()`，避免后台播放

##### 12.3.7.3 跳到下一张逻辑

| 场景 | 行为 |
|------|------|
| 手动 ←/→/点缩略图 | 立刻跳（视频被截断）|
| 视频自动 `ended` 事件 | **放完自动跳下一张**（Apple Photos 行为）|
| 幻灯片模式 | 视频按真实时长停留，图片按 3s 停留 |
| 边界 | 第 1 张 ← → 跳到最后；最后一张 → 跳到第 1 张（环形）|

##### 12.3.7.4 缩略图条：混排

- 同一滚动条内**图片/视频缩略图混排**，不分组
- 视频缩略图带**双重标识**：

  - 左下角**时长标签**（`0:24` / `1:34` / `2:48`，等宽字体，黑底半透）
- 点击缩略图切换主图区内容
- 缩略图条自适应 `current` 居中（`scrollIntoView({ inline: 'center' })`）

### 13.4 v0.7 数据目录变更（解决"升级丢失数据"问题）

> **v0.6 问题**：用户反馈每次更新 exe 后，数据库和配置文件被清空。
> **根因**：当前 `_get_app_data_dir()` 在打包模式下返回 `<exe>/`，升级覆盖 exe 目录时数据被一并清空。

**v0.7 决策**：把数据目录从 `<exe>/.config/` 改为 **`%USERPROFILE%/Documents/BlurArc/`**（我的文档）。

| 项 | v0.6（当前）| v0.7（变更后）|
|----|------------|---------------|
| 数据库 | `<exe>/.config/photo_manager.db` | `%USERPROFILE%/Documents/BlurArc/.config/photo_manager.db` |
| 配置文件 | `<exe>/.config/config.json` | `%USERPROFILE%/Documents/BlurArc/.config/config.json` |
| 缩略图缓存 | `~/.photomanager/thumbnails/` | `~/Documents/BlurArc/thumbnails/` |
| 手机上传临时 | `<exe>/.config/phone_upload/` | `~/Documents/BlurArc/.config/phone_upload/` |
| 手机配对 tokens | `<exe>/.config/mobile_tokens.json` | `~/Documents/BlurArc/.config/mobile_tokens.json` |
| 用户配置 | `~/.photo_organizer_config.json` | `~/Documents/BlurArc/config.json`（合并）|

**目录结构**：

> **原则**：**所有用户数据**（数据库、配置、缩略图、临时文件、日志、移动设备数据）**统一**放在 `~/Documents/BlurArc/` 下，**不分散到系统其他位置**。升级/卸载只动 exe 目录。

```
C:\Users\Alice\Documents\BlurArc\          ← 根：所有用户数据
├── .config\
│   ├── photo_manager.db                   # 主数据库
│   ├── config.json                        # 应用配置
│   ├── mobile_tokens.json                 # 手机配对 token
│   ├── devices.json                       # 移动设备配对记录
│   └── phone_upload\                      # 手机上传临时目录
├── thumbnails\                            # 缩略图缓存（跨相册共享）
├── cache\                                 # 其他缓存（视频预览/AI 推理等，v0.7 预留）
├── logs\                                  # 应用日志
└── exports\                               # 导出文件（v0.8+）
```

**所有数据集中**（不要分散）：

| 数据类型 | 路径 | 备注 |
|----------|------|------|
| 数据库 | `~/Documents/BlurArc/.config/photo_manager.db` | 必存 |
| 应用配置 | `~/Documents/BlurArc/.config/config.json` | 必存 |
| 缩略图缓存 | `~/Documents/BlurArc/thumbnails/` | 可重建（删了不影响数据）|
| 手机配对 | `~/Documents/BlurArc/.config/mobile_tokens.json` | 必存 |
| 手机上传 | `~/Documents/BlurArc/.config/phone_upload/` | 必存（导入完清理）|
| 应用日志 | `~/Documents/BlurArc/logs/` | 可删（出问题排查用）|
| 缓存（v0.7 预留）| `~/Documents/BlurArc/cache/` | 视频预览帧等 |
| 导出（v0.8+）| `~/Documents/BlurArc/exports/` | 用户导出的照片 |

**❌ v0.6 分散位置（v0.7 全部收回）**：
- `~/.photomanager/thumbnails/` → 改为 `~/Documents/BlurArc/thumbnails/`
- `~/.photo_organizer_config.json` → 改为 `~/Documents/BlurArc/.config/config.json`
- `<exe>/.config/...` → 改为 `~/Documents/BlurArc/.config/...`
- 系统临时目录（`%TEMP%/blurarc_*`）→ 改为 `~/Documents/BlurArc/cache/`

**为什么选"我的文档"？**
- ✅ **用户能直接看到**（不用记 `%APPDATA%` 这种隐藏路径）
- ✅ **系统重装/迁移方便**（"我的文档"会被备份软件优先备份）
- ✅ **卸载时不会误删**（卸载 exe 不影响 Documents）
- ✅ **跨平台一致**（macOS = `~/Documents/BlurArc/`，Linux = `~/Documents/BlurArc/`）

**平台差异**：
| OS | 路径 |
|----|------|
| Windows | `C:\Users\<user>\Documents\BlurArc\` |
| macOS | `/Users/<user>/Documents/BlurArc/` |
| Linux | `/home/<user>/Documents/BlurArc/` |

**不迁移**（v0.7 决策）：
- ❌ v0.6 用户的旧数据**不自动迁移**
- ✅ 在 v0.6.1 升级提示里说明："v0.7 起数据存到我的文档，需要重新导入"
- ✅ v0.7 首次启动检测到旧路径 `<exe>/.config/` 时，**只 toast 提示**，不自动复制

## 14. 跨章约束

### 14.1 平台支持

**v0.7 仅支持 Windows 桌面端**（PyWebView + Flask）。

涉及 Windows 专属 API 的功能：
- "在资源管理器中显示"：`subprocess.Popen(['explorer', '/select,', path])`
- FFmpeg 路径：`backend/ffmpeg_binaries/ffmpeg.exe`
- DB 路径：v0.7 起改为 `~/Documents/BlurArc/.config/`（详见 §13.4）

未来 Mac / Linux 适配时，这些功能需要重写或条件分支（v0.7 不预留抽象层）。

### 14.2 "导出"功能范围

v0.7 "导出..." 菜单仅支持 2 种粒度：

| 导出类型 | 范围 | 格式 | 实现 |
|---------|------|------|------|
| 导出原图 | 单张 / 批量 | 原始文件复制到目标目录 | 客户端 `shutil.copy2` + 选择目录 |
| 导出缩略图包 | 批量 | 缩略图打包为 ZIP | 复用 `thumbnail_manager` 现有缩略图 |

**不支持**（v0.7 范围外）：
- ❌ 视频导出
- ❌ 调色 / 滤镜 / 旋转后再导出
- ❌ HEIC → JPG 格式转换
- ❌ 邮件 / 网盘 / 第三方上传

### 14.3 性能预算

| 场景 | 目标 |
|------|------|
| 收藏视图首屏（10K 收藏） | < 500ms |
| 单个相册内照片列表（1K 照片） | < 300ms |
| "不在相册中" 过滤（20K 照片） | < 1s（**必须**用 LEFT JOIN，见 §4.4） |
| 时间线无限滚动每次加载 | < 800ms（12 月） |
| 加入收藏 / 取消收藏 API 响应 | < 100ms |
| 加入相册 modal 打开 | < 200ms |

### 14.4 国际化

v0.7 仅做**简体中文** UI（zh-CN）。
- 文案集中在 `frontend/src/i18n/zh-CN.ts`（新增文件）
- 不引入 i18n 框架（v0.7 直接 hardcode）
- 时间格式：`YYYY-MM-DD` / `YYYY年M月D日`
