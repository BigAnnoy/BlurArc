## 关键决策与权衡

### 1. PairingService / UploadService 不存在 → 测试 ApiClient

实际项目把配对和上传逻辑都内联在 `ApiClient` 中（`pairingRequest` / `getPairingStatus` / `submitPairingCode` / `cancelPairing` / `uploadFile`）。重新创建 `PairingService` / `UploadService` 类会破坏现有架构且超出 Plan E 范围。改为测试 `ApiClient` 的对应方法。

### 2. dio_mock 不引入

计划提议 `dio_mock: ^0.0.1`，但 `ApiClient` 在构造函数中实例化 `Dio`，没有 setter 注入点。要 mock Dio 需要重构 `ApiClient`（影响生产代码）。Plan E 范围是只增/改 test/，所以改用 mocktail 直接 mock `ApiClient` 类本身（用于 widget 测试），服务层测试则用真实 `ApiClient` + 临时 SharedPreferences mock（验证 URL 生成、SharedPreferences 持久化、状态管理等不依赖真实 HTTP 的逻辑）。

### 3. 页面层部分测试依赖真实 ApiClient

`HomePage` 的 disconnect 测试需要真实 `ApiClient`（因为 `HomePage.initState` 覆盖 `api.onDisconnected` 回调，mock 时该赋值是 no-op）。改用真实 `ApiClient` + `setConnectionParams('127.0.0.1', 8900)` 来观察回调触发。

### 4. widget_test.dart 改为渲染 + 状态断言

原 `find.text('Blur Arc')` 期望 AppBar 标题里有 "Blur Arc" 文字，但实际项目 AppBar 标题是 `BlurArcLogoWithText`（PNG 图片）。改为：
- 断言 `MaterialApp` / `Scaffold` / `BlurArcLogoWithText` 存在
- 断言 ConnectScreen 状态文本（"正在扫描..." / "未找到 Blur Arc 服务..." / "步骤 1/2：连接设备"）至少一个存在
